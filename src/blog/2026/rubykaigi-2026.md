# RubyKaigi 2026

今年の4月22-24日の3日間、RubyKaigi[^rubykaigi-2026]に行ってきた。

場所は函館。北海道は人生で2回目だが、函館は初訪問。

実は昨年も参加していたが、色々あってブログをサボってしまった。今年は何とか書く。GW休みも取れたし。

## Day -1 — Mon

羽田から函館空港へ。ホテルに着いた早々、ケータイの緊急地震速報が鳴る[^earthquake]。しばらく館内にとどまるようアナウンス。津波警報出ていたし、実際に来たらしい（影響なかったが）。

しばらく部屋でビクビク様子を見ていたが、お腹が空いたので思い切って外出。コンビニくらい空いてるだろう。しかし町は意外と平穏だった。適当に見つけた近くの居酒屋で夕食を済ます。

## Day 0 — Tue

会社のスポンサーイベントで市電を貸し切ることになっていたので、朝から電停周辺を下見する。

その後、朝市へ。海鮮丼を食す。まあ美味い、安い。

それから夕方まで時間があったので、函館山周辺の洋館街〜造船所付近をブラブラする。
雰囲気は少し故郷の長崎に似ている。異国情緒。しかし道路幅が全然違う。多分3倍くらい広いし、直線的。
長崎は狭い地形だからギチギチに建物があり、街が何だか忙しない。ただ歩く距離は節約できる。坂や階段は多いが。

夕方に会場へ行き、会社のSwagの到着を確認。同僚と軽く明日の確認。
ちなみにホテルは函館駅前で、会場の函館アリーナまで市電でおよそ30分くらい。

## Day 1 — Wed

朝は会社の市電貸切イベントの手伝い。下見していたとはいえ初日だったので、多少混乱した。
幸い大きなトラブルはなかったが、翌日以降のオペレーションを少し見直すことに。翌日からは市電側で人員を手配してもらい、大いに助けられた。

RubyKaigi 2026のスタートは、tagomorisさんによるRuby Box[^ruby-box]キーノート。
個人的には、Ruby Boxは大規模モノリスにおけるスマートな解決策にならないものかと期待している。
そもそもデカいソフトウェアをどう分割するかという問題は難しい。モノリスvsマイクロサービス。組織分割という人間の問題も関わる。技術的に解決できるとスマートなのだが[^module-system]。

jimlockさんのDateライブラリ再実装の話も良かった。標準ライブラリがどんな保守問題を抱ええていて、それをどうやって解決していくか具体的なエピソードや時系列で語られていて、めちゃくちゃ分かりやすかった。
こういった取り組みは、ずっとRuby本体の動向をウォッチしてないとなかなか難しい。メンテナ不足とはいえ、安易に貢献を受け入れられるかといったらそれは別問題で、長期間熱量を維持しつつ信頼関係も構築する必要がある。

osyoyuさんのプロファイラの話も興味深かった。正直あまり詳しくはないが、仕事上Datadogをかなりヘビー使ってて、最近興味が出てきた分野。
今回の話では既存のプロファイラ実装の限界を示しつつ、Rubyが提供する新プロファイラAPIを提案してた[^profiler]。

kazuhoさんの話も良かった。半分くらいHTTP/3の性能改善の話なんだけど、RFC仕様に出てくるレベルなので、なかなか他では聞けない内容だったと思う。
あとの半分は、jqが遅いから代わりに自作したjrf[^jrf]というツールの話。もちろんRuby実装。jqのクエリ言語は覚えにくいからRubyで書けばいいじゃんという発想。しかもjqより速いし巨大JSONデータも処理できる。

最後にspikeolafさんの話が印象に残った。Prismの保守性に優れているという主張を丹念に検証していく。
特にPrismが採用しているPratt Parsingアルゴリズム[^prism]について。手書きプログラムだから保守性が高い、逆に文法ファイルは保守性が低い、とは単純に言えないのではないか。
Ruby文法に対応するため、PrismはPratt Parsingを拡張する必要があり、文法への深い理解が必須。文法理解の必要性は変わらない。

今後ますますLLMに文法ファイルを読み書きさせるようになる…と考えると、衝突検出などの機構をもってるLR法がむしろ有利になるかもしれない。「文法」なんだから形式言語で記述されるべきだよね、アドホックに実装すべきものではないよね、という主張はなるほど理に適っている。
いやむしろ、プログラミング言語の文法は形式言語で記述されるべきかも。実用上Prismの功績は大きいけど、手書きはあくまで過渡期のものであって、理想はそこじゃないでしょ、みたいな気持ちがある。スピーカーの熱量に感化されたのかもしれない。

ところで、Official Partyで久しぶりに元同僚のsumyappさんに出会った。あまりに懐かしかったので、近々同窓会を企画しようと思っている。

帰り際、偶然見かけたHono作者のyusukebeさんに声をかけさせてもらった。数日前に出てたインタビュー記事[^hono-article]を読んでいて、めちゃくちゃ共感していたのだった。
「プルリクの意味が変わってしまった」と話していたのが記憶に残っている。これまでOSSに内在していた未解決問題[^dependency-meme]がAI/LLMの暴力で表面化された。防御的な反応が出るのは当然な一方、どう向き合えばいいのか。個人的には趣味で楽しくOSS活動していきたい。
長く続けるには「楽しさ」が秘訣だと思う一方、いかにそれを守るかが問われてる。誰だってAI生成の雑なコードをレビューしたくないし、脆弱性報告で疲弊したくない。

## Day 2 — Thu

2日目は聴くペースを落としつつ。なぜか年々聴く体力・集中力が落ちている気がする。選択と集中。
あとで評判を聞くと「やっぱり聴いておけば良かった」と後悔するトークもあったけど、こればかりは仕方がないと自分に言い聞かせる。

本日最後のst0012さんのRDocトークは、発表から楽しみにしていたものの一つ。というのも、RDocで生成されるドキュメンテーションサイトにはかなり前から不満があり、ときどきPRを投げていたから[^rdoc-contribution]。

去年の12月にAkikiという新テーマがデフォルトになり、[docs.ruby-lang.org][docs-ruby]での体験がめちゃくちゃ良くなったのはst0012さんのおかげ[^aliki-theme]。サイトの操作性・利便性向上はもちろん、SEOで検索上位にヒットするようになった。今まで公式サイトを見つけるに苦労してたので本当にうれしい。

一つ驚いたのが、RBSサポートの話[^rdoc-rbs]。例えば`Array#count`は`:call-seq:`ディレクティブ[^call-seq]を使って以下のように定義されており[^ruby-array]、HTMLドキュメントもその通りに生成される。

```
count -> integer
count(object) -> integer
count {|element| ... } -> integer
```

一方、RBSでは以下のように定義されており[^rbs-array]、以前からチグハグさが気にはなっていた。

```rbs
() -> ::Integer
(Elem obj) -> ::Integer
() { (Elem) -> boolish } -> ::Integer
```

今のところは両方を生成するようになりそうだが、将来的にはRBSの方に寄せることになると思う。プレーンテキストよりも形式言語で記述する方が正確だし、ツールの恩恵も得られるから。今後に期待。

全セッション終了後は、会社のスポンサーイベントで市電貸切Drink-upを開催。夜の函館市内を走る市電に乗って飲み食いするという、なかなか得難い経験をした。普通の飲食店でのそれとは違い、変にテンションが上がった。市電という優遇された交通機関内で合法的に飲食をするという背徳感（？）のなせるワザだろうか。できるならばもう一度体験したい。

## Day 3 — Fri

最終日、なぜかすごく疲れており、あまりトークの記憶がない。いくつか聞きたいと思っていたものも聞き逃してしまった。今回は例年よりトーク外の行動が活発だったからだろう。その分より多くの「廊下トーク」をしたんだと自分に言い聞かせる。

ところどころ記憶が断片的だが「Committers and the World」でko1さんが極めて重要と同時にセンシティブなトピックをもってきたのが頭に残っている。何が起こるかはわからないけれど、Rubyコミュニティにはかなり信頼を置いているから、意外と何とかなるじゃないかと思っている。何らかの文書化は必要になってくるとは思うけど、信頼がなければ空文化するだけだから。

脱線してしまうが、「良いコミュニティ」⇆「良いソフトウェア」の相互作用が成立するんだと確信している。そして、コミュニティの方が作るのがずっと難しい。一見対照的なものが、実は相互に依存ているという点が面白い。時系列上はソフトウェア→コミュニティの順なのだが。

riseshiaさんの型に関するトークも印象に残っている。2日目のmameさんのトークも似ていて、どちらもできるだけ型を書かずに（つまり推論で）**実用上便利な**型チェックをどう実現するかにフォーカスしている。riseshiaさんのトークでは、現在の各型ツール（Steep/Sorbet/TypeProf/Ruby LSPなど）の位置付けをグラフ化しているのがわかりやすかった。Rubyでの型普及を阻害する要因を把握するのに便利だ。

多分、型を書くインセンティブがどうしても弱くなってしまうのが、普及を阻害する要因の一つ。Railsをメインターゲットに置くとして、メタプロで生成される大量の型をどう扱うか。各ツールがなかなか決定版になりえてない。「実用上便利」を達成しないと普及しないが、その「実用上便利」にツールでブレがある。推論でカバーするのか、型生成で質を確保するのか。
大量に生成される型に対する保守コストなど。テストを書くインセンティブは確立されているので、それで型で検出できる問題がカバーできてしまうという側面もある。

インラインRBSはメンテコストを下げるのか、いやむしろコードの認知コストまたはLLMによるトークン消費コストを上げるのか。別ファイルにある方が消費コストは上がるかもしれないし、LLMのことを考え出すとキリがない気もする。来年にはまた変わってるだろうし。

Railsアプリを書く側からすると型付けインセンティブが低いか、ライブラリ・フレームワークが型を提供しているとめっちゃ便利というのは間違いなくある。CIだけじゃなく、普通にコードを書く上でも。Claude Codeなどをターミナルでメインで使うようになってからVSCodeを開く機会がほぼなくなってしまったけど、Ruby LSPはClaudeが裏でめっちゃ使ってるし。
そうそう、エディタ上で定義ジャンプしたときに、インラインRBSだと実装と一緒に型・ドキュメントがすぐに確認できるのが非常に良い。できれば標準ライブラリはすべてインラインRBSになってほしいなぁ。

3rd-party gemに関してはやはりコミュニティベースの努力が依然として必要で[^gem-rbs-collection]、そこはどうしてもインセンティブが低くなりがち。RBS同梱してくれるgemも徐々に増えてきているとは思うけど（AWS SDKとか）。Gemfileに`gem "rails" ; gem "@types/rails"`と書けるようになると少しは便利になるかも。そういえばRubyGemsにネームスペースを入れる話があったような。

徐々に進化してきているので、今後に期待。なんかブレイクスルーがあると一気に普及する気もする。

MatzキーノートはSpinel[^spinel]一色に。Rubyネイティブ実行は昔からの夢。HomebrewやFluentdなんかは配布が楽になるのではないだろうか。Go製のCLIツールなんかも一部Rubyでリライトされたりするんじゃなかろうか。

セッションすべて終了後は、特にイベントには参加せず。riseshiaさんとのんびり話しながら、ウニ、函館山夜景を堪能。疲れていたので、まったり過ごせて大変良かった。当分あの質・量のウニは食べられないだろうなぁ。

## Day 4 — Sat

疲れを見越して日曜まで滞在を予定していて正解。心残りは前日夜までの五稜郭ライトアップを見逃したこと。日中に行きかけたが、あまりの混雑に断念した。ちょうど開花シーズンだった。

思い立ってレンタカーを借りて、新千歳までドライブすることにした。大沼公園で駒ヶ岳を望む。八雲町のSAで内浦湾と遠く羊蹄山を眺める。洞爺湖では羊蹄山がよく見えた。登別のゲストハウスで宿泊。クマ牧場は断念。

景色は最高だったが、高速が1車線の区間が長くしかも結構みんな飛ばすので、かなり神経を使った。直線が多く道路が広いので運転はしやすかったが。一般道でも普通に80とか出すのでビビった。

## Day 5 — Sun

白老のウポポイ[^upopoy]を訪れる。ゴールデンカムイとかで触れる機会が増えたけど、そう言えばアイヌのことほとんど知らないなと思っていて、一度来てみたかった。
北海道の地名の8-9割はアイヌ語由来なのだそうだ。例えば登別は「ヌプㇽペッ（nupur-pet）」（色の濃い・川）に由来する[^noboribetsu]。

また、アイヌ語はどの語族にも属さない「孤立した言語」で、文字をもたないんだそう。Rubyはプログラミング言語で、もちろん自然言語とは違うんだけど、言語って本当面白いなぁと思う。パーサだったりUnicodeだったりLLMだったり、RubyKaigiが言語理解を深めるきっかけになったのは間違いない。

アイヌ音楽の演奏も聴いたけど、何か脳みそのいつもと違う部分がザワザワする感覚。普段触れてる・見慣れてる世界ってずいぶん狭いんだなと感じた。

途中ウトナイ湖に少し立ち寄って、新千歳から帰京。疲れたけど、なんか充実してた。

## Bookstore

店長のko1さんにも直接伝えたが、本屋さんが年々拡充していて素晴らしい。選書センスに安心できるし、表紙見てるだけで楽しい。昨年1冊、今年は3冊。昨年売り切れだった『白と黒のとびら』にリベンジできた。昨年買った『自動人形の城』が面白く、同じ著者の川添愛さんの本がまた並んでいたので衝動買いしてしまった。

あとの2冊は『型システムのしくみ』と『Fluentd実践入門』。元々欲しいと思っていたんだけど、今回店頭に見つけて即決。著者のmameさんとtagomorisさんそれぞれからサインもいただけた。サインがあるといつもより読むモチベが上がる気がする。

## Wrap-up

RubyKaigi 2018以来、現地参加は6回目なんだけど、ようやく自分のペースでの参加の仕方をつかんだ気がする（単に体力が落ちてる気もするが）。振り返ると、初日で体力をかなり使ってるかもしれない。ペース配分大事。

北海道・函館も大変良かった。地震で一時はどうなることかと思ったけど、無事終わって本当に良かった。ブログもなんとか残せたし。

RubyKaigiは1年でガラッとAI/LLMが話題の中心になった感があるけど、来年は反動が来そうな気もする。どうだろう。

さて次回は宮崎。（同じ九州・長崎出身にもかかわらず）未踏の地なので楽しみ。
